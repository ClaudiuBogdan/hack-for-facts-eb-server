# Campaign Admin User Interactions Audit API

**Status**: Accepted
**Date**: 2026-04-11
**Author**: Codex

## Problem

The first campaign-admin API iteration exposed only a narrow review queue for a
small allowlist of Funky interactions. That no longer matches product needs.

Campaign admins now need one safe admin surface that can:

- audit the full set of campaign interaction records generated by the civic
  module
- keep review actions available only for records that truly belong to an async
  review workflow
- avoid turning the endpoint into a raw `UserInteractions` dump

The main gap in the earlier design was that it excluded valid campaign records
such as:

- budget document, publication date, status, contact, contestation
- participation report
- persisted civic quiz answers
- self-send public-debate submissions that are part of the campaign audit trail
  but not server-reviewable

## Decision

Keep the existing privileged boundary and safe-projection approach, but expand
the surface from a review queue into a campaign interaction audit API with a
reviewable subset.

## Boundaries

- Auth stays session-based through `requireAuth(...)`.
- Authorization stays Clerk-permission-based with `campaign:funky_admin`.
- Fail-closed startup and request behavior stays unchanged.
- Raw `record` and `audit_events` are never returned to the admin client.
- The public-debate approval side effect path stays unchanged for approved
  `request_platform` submissions.

## Model

Each campaign interaction is registered in server code with:

- `interactionId`
- label
- projection kind
- step location when known
- `adminAuditVisible`
- `reviewable`
- optional reviewable submission-path restrictions
- optional institution-thread-summary support

Two distinct concepts now exist in the same admin surface:

1. Audit visibility
2. Reviewability

These are intentionally not the same.

## Funky Inventory

### Reviewable interactions

- `funky:interaction:city_hall_website`
- `funky:interaction:budget_document`
- `funky:interaction:budget_publication_date`
- `funky:interaction:budget_status`
- `funky:interaction:city_hall_contact`
- `funky:interaction:budget_contestation`
- `funky:interaction:public_debate_request` only when
  `submissionPath = request_platform`

### Audit-only interactions

- `funky:interaction:funky_participation`
- `funky:interaction:public_debate_request` when
  `submissionPath = send_yourself`
- `ch-civic-01-how-module-works-q1`
- `ch-civic-01-why-budget-matters-q1`
- `ch-civic-01-what-campaign-is-q1`
- `ch-civic-02-cycle-q1`
- `ch-civic-02-deadlines-q1`
- `ch-civic-02-rights-q1`
- `ch-civic-03-why-check-q1`
- `ch-civic-04-why-debate-q1`
- `ch-civic-04-next-steps-q1`
- `ch-civic-05-preparation-q1`
- `ch-civic-06-when-contest-q1`

### Explicit non-inventory findings

- `MarkComplete` persists lesson content progress, not `UserInteractions`, so it
  is outside this API.
- `ExpandableHint` does not persist interaction records.
- `CivicModuleShareCta` does not persist interaction records.

## Response Semantics

### Audit list

`GET /api/v1/admin/campaigns/:campaignKey/user-interactions` is now an audit
list, not a pending-review queue.

The list returns all configured audit-visible interaction rows that match the
filters. It includes a per-row `reviewable` boolean so the client can render
review actions only where allowed.

### Review fields

- `reviewStatus` is derived only for reviewable rows.
- `pendingReason` is derived only for reviewable pending rows.
- Audit-only rows keep their real lifecycle `phase`, but `reviewStatus` and
  `pendingReason` stay `null`.
- The review write endpoint still rejects audit-only rows.

This prevents immediate, static, or non-reviewable-path records from being made
to look like queue items.

## Safe Projections

The admin API remains projection-based. No raw user-controlled JSON blobs are
returned by default.

Current projections:

- `public_debate_request`
  - `institutionEmail`
  - `organizationName`
  - `submissionPath`
  - `isNgo`
- `website_url`
  - `websiteUrl`
- `budget_document`
  - `documentUrl`
  - `documentTypes`
- `budget_publication_date`
  - `publicationDate`
  - `sources[] { type, url }`
- `budget_status`
  - `isPublished`
  - `budgetStage`
- `city_hall_contact`
  - `email`
  - `phone`
- `participation_report`
  - `debateTookPlace`
  - `approximateAttendees`
  - `citizensAllowedToSpeak`
  - `citizenInputsRecorded`
  - `observations`
- `contestation`
  - `contestedItem`
  - `reasoning`
  - `impact`
  - `proposedChange`
  - `senderName`
  - `submissionPath`
  - `institutionEmail`
- `quiz`
  - `selectedOptionId`
  - `outcome`
  - `score`

The quiz projection is intentionally minimal. It exposes the authored option id
and evaluated outcome without leaking full interaction payloads or audit-event
blobs.

## Metadata

`GET /api/v1/admin/campaigns/:campaignKey/user-interactions/meta` now reflects
the full audit inventory and marks whether each interaction type is reviewable.

Stats are computed across all audit-visible rows. Review-status counts are still
derived only from the reviewable subset.

## Thread and Risk Semantics

- Institution-thread summary remains available only for interaction types that
  support it.
- For Funky today that means public-debate records on the reviewable
  `request_platform` path.
- Self-send debate-request rows remain visible for audit, but they do not expose
  review-derived thread or pending-risk semantics.

## Review Endpoint

`POST /api/v1/admin/campaigns/:campaignKey/user-interactions/reviews`

- accepts only reviewable rows
- continues to reject audit-only rows with fail-closed lookup behavior
- preserves existing public-debate approval side effects

## Why The Old Allowlist Changed

The earlier narrow allowlist solved the immediate security concern but created a
product mismatch: admins could review a subset of campaign submissions while
remaining blind to the rest of the campaign’s persisted user interaction trail.

The new design expands visibility without widening the data boundary:

- the route is still campaign-scoped
- only server-registered interaction ids are returned
- every returned row uses an explicit safe projection
- review actions stay limited to the subset that actually supports them

## Consequences

- Admins can audit the full Funky civic interaction trail in one place.
- The client can render review controls conditionally from server data.
- Immediate and quiz interactions remain visible without being forced into an
  async-review workflow.
- New campaign interactions must still be added intentionally in server code
  with an explicit projection before they appear in this privileged API.
